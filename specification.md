Project Specification: React Core (Component, State, Hooks) 학습용 직접 구현

1.  프로젝트 개요 및 목적
    이 프로젝트의 핵심은 React의 핵심 동작 원리인 Component, State, Hooks를 JavaScript or TypeScript로 직접 구현하며 그 이면의 메커니즘을 깊이 있게 학습하는 것입니다. 지난주에 구현한 Virtual DOM과 Diff 알고리즘을 연동하여 실제 프레임워크가 어떻게 화면을 업데이트하는지 체득합니다.

    최종 목표: 구현된 코드를 바탕으로 Top-down 방식의 학습을 수행하며, "함수는 매번 새로 실행되는데, 상태는 어떻게 유지할까?"라는 핵심 질문에 대해 기술 면접에서 완벽하게 설명할 수 있는 수준에 도달함.

    에이전트의 역할: 단순히 작동하는 코드를 주거나 내 대신 생각하는 것이 아니라, 내가 더 나은 사고를 할 수 있도록 돕는 도구의 역할을 수행할 것.

    코드 제공 시 반드시 **'왜(Why)' 이 방식을 선택했는지, '작동 원리', '장단점(Pros/Cons)'**을 코드 내 주석과 설명으로 상세히 제공할 것.

    나중에 직접 구현할 수 있도록 논리적 흐름이 명확한 코드를 작성할 것.

    한 줄 한 줄마다 코드 작동 원리를 주석으로 적을 것.

2.  시스템 가이드라인 (에이전트 필수 준수 사항)
    상세 주석 (Deep Documentation): 모든 핵심 함수와 로직에 다음 내용을 주석으로 포함하세요.

    @description: 이 함수/로직이 해결하려는 문제.

    @logic: 내부 작동 원리 (단계별 설명).

    @pros_cons: 이 구현 방식의 장점과 단점 (특히 메모리나 성능 관점에서).

    @interview_tip: 이 부분과 관련해 면접에서 나올 수 있는 질문과 답변 포인트 (Closure, Lifting State Up 등).

    기술 스택: 외부 프레임워크(React, Vue 등) 없이 Vanilla Javascript (또는 Typescript), HTML, CSS만 사용합니다.

    코드 품질: 가독성을 최우선으로 하며, 작동 원리를 명확히 보여주는 정갈한 코드를 유지합니다.

3.  구현 상세 요구사항
    3-1. Component (컴포넌트) 아키텍처
    함수형 컴포넌트: UI를 나누는 작은 단위로, 각각의 컴포넌트는 자신만의 로직과 렌더링을 담당하며 반드시 함수형으로 구현합니다.

        FunctionComponent 클래스: 함수형 컴포넌트를 감싸고 관리하는 클래스를 직접 구현합니다.

        hooks: 상태 저장을 위한 배열 (클로저 및 인덱스 활용).

        mount(): 초기 렌더링을 수행하는 메서드.

        update(): 상태 변경 시 호출되어 다시 렌더링을 수행하는 메서드.

    3-2. State (상태) 관리 및 제약 조건
    상태의 정의: 값이 변경될 수 있는 동적 데이터이며, State가 바뀌면 화면이 자동으로 다시 그려져야 합니다.

        최상위 상태 관리 (Lifting State Up): State(상태)는 오직 "최상위 컴포넌트(루트 컴포넌트)"에서만 관리합니다.

        Stateless 자식 컴포넌트: 자식 컴포넌트는 State를 가지지 않으며, 부모로부터 전달받은 props만 사용하여 화면을 그리는 순수 함수(Pure Function)로 구현해야 합니다.

    3-3. Hooks 구현
    함수형 컴포넌트에서도 상태와 생명주기를 사용할 수 있게 해주는 다음 3가지 핵심 Hook을 구현합니다. (Hook은 최상위 컴포넌트에서만 사용 가능합니다.)

        useState: 상태 값과 그 값을 갱신하는 함수를 반환.

        useEffect: 렌더링 이후 외부 시스템과의 동기화 및 사이드 이펙트 처리.

        useMemo: 계산 비용이 높은 연산의 결괏값을 메모이제이션.

    3-4. Virtual DOM 연동 (Week 3 통합)
    지난주에 만든 VDOM, Diff, Patch 모듈을 활용하여 렌더링 파이프라인을 구축합니다.

        동작 흐름: 1. 상태 변경 시 새로운 Virtual DOM 생성 2. 이전 Virtual DOM과 비교 (Diff) 3. 바뀐 부분만 최소한으로 찾아내어 실제 DOM에 반영 (Patch)

        목표: 상태가 변경될 때마다 화면 전체를 다시 그리지 않고 필요한 부분만 업데이트되도록 합니다.

4.  중점 학습 및 인터뷰 체크리스트
    에이전트는 아래 개념들이 코드 구현에 어떻게 녹아있는지 명확히 설명해야 합니다.

    Hooks의 비밀: 실행이 끝난 함수의 상태가 어떻게 유지되는가? (Closure와 배열 인덱스를 활용한 Hook의 작동 원리)

    상태 끌어올리기 (Lifting State Up): 루트 컴포넌트에서만 상태를 관리하고 자식은 Props만 받을 때의 아키텍처적 장단점과 단방향 데이터 흐름.

    순수 함수와 UI: 컴포넌트를 상태 없는 순수 함수로 만들었을 때 얻을 수 있는 이점(테스트 용이성, 예측 가능성).

    렌더링 최적화: useMemo와 Virtual DOM의 Diff 알고리즘이 결합하여 불필요한 연산과 DOM 조작을 어떻게 방지하는가?

    생명주기 관리: FunctionComponent의 mount와 update 과정에서 Hook의 초기화와 재실행 로직 차이.

5.  단계별 실행 계획 (Action Plan)
    Step 1: 프로젝트 폴더 구조 세팅 및 Week 3의 VDOM/Diff/Patch 모듈 연동.

    Step 2: FunctionComponent 클래스 설계 및 순수 함수형 자식 컴포넌트 렌더링 구조 구현 (mount, update 뼈대).

    Step 3: 최상위 컴포넌트에 적용할 useState 구현 및 상태 변경에 따른 재렌더링 흐름 완성.

    Step 4: useEffect와 useMemo를 구현하여 Hook 배열 시스템 확장.

    Step 5: 구현한 Core 시스템을 활용하여 루트 컴포넌트에서 상태를 관리하고 하위로 전달하는 데모 웹 페이지 제작 및 리팩토링.
